Production-grade · 4 services · event-driven

A multi-tenant marketplace built like infrastructure, not a side project.

Next.js 16 storefront with React 19 and the React Compiler, three serverless AWS microservices for chat, search, and transactional email, all communicating over an SNS → SQS → Lambda backbone. Multi-tenant from the schema up; multi-currency, multilingual, S3-backed media, Stripe + COD checkout.

01 - Storefront

The landing page does the marketing

Server-rendered marketing surface with a featured-departments rail. The Shop by Department carousel is fed from the same Category tree that powers product filtering - one source of truth, two surfaces. Theme, currency, and locale switchers are all reflected server-side on the very first render - no client-side hydration flash, no layout shift.

02 - Discovery

Faceted listing with a frictionless add-to-cart

Filter sidebar (department, price, customer reviews, deals, product type, brand) keeps all state in the URL via nuqs - every filter combination is shareable and SSR-correct. Cards surface aggregate ratings, per-product video badges, and compare-at-price strikethroughs so buyers can compare without a click-through. The Amazon-style quick-add modal lets them pick a variant and add to cart without leaving the grid; the search blob is denormalized across all four locales so "haljina" finds an English-only product.

03 - Product detail

Variant matrix, reviews, and a buyer-to-seller chat hook

Two-axis variant matrix (color × size) with per-variant SKU, price, stock, and dedicated images. The compare-at price strikethrough and per-variant discount badges are computed server-side. The Message seller button opens the WebSocket chat drawer scoped to a fresh buyer↔seller conversation for that product. Below the fold: a per-product review list with star ratings, then three Embla autoplay carousels off one shared ProductCard - You may also like (brand & category affinity), Frequently bought together (derived from real OrderItem co-occurrence), and Recently viewed (replayed from an append-only interaction log keyed by signed-in user or anonymous session).

04 - Real-time chat · marketplace-messaging

WebSocket chat built on Lambda & DynamoDB

Buyers and sellers talk in real time over a persistent WebSocket managed by API Gateway. All state - conversations, messages, connection records, reactions - lives in a single DynamoDB table with two GSIs. Fan-out delivery queries GSI2 for every active connection of the recipient and pushes via ApiGatewayManagementApi. Image and PDF attachments go browser → S3 via presigned PUTs (zero-byte Lambda traffic); the in-app PDF viewer is rendered with pdfjs-dist.

05 - From checkout to inbox · marketplace-notifications

One purchase, four side-effects, zero coupling

The Stripe webhook persists the order and publishes a single SNS event, then returns 200. The notifications Lambda consumes the event from SQS, claims it idempotently in DynamoDB, hydrates with order details via an internal API, and dispatches localized SES emails - one to the buyer, one per seller organization. The marketplace itself never touches SES. The seller sees the same order in the dashboard, ready to fulfill. The same fire-and-forget pipeline carries the whole post-purchase lifecycle - shipping & delivery, Stripe Connect payouts, returns/RMA, and AI review-moderation decisions - each a typed event, rendered in the recipient's language. Every order also gets a sequential PDF invoice rendered server-side with @react-pdf/renderer (line items, product thumbnails, applied coupon) and stored on S3.

06 - Admin · bulk import

CSV bulk import with per-row validation

Admin power-feature: download a template, paste or upload a CSV, get a row-by-row preview with valid / invalid badges and the exact Zod error message inline. Brands and categories are resolved by slug. The importer reuses the same server actions and validation schemas as the single-product form - no duplicate validation logic.

07 - i18n · currencies · theming

Four locales, three currencies, themable storefront

next-intl v4 powers four locales (en, sr, de, es). Per-product translations live in a dedicated table; the search blob is rebuilt across all locales so cross-language search works out of the box. Display currency switches between USD, EUR, and RSD - rates are refreshed weekly by an EventBridge cron in the notifications service. Four themes ship out of the box - light, dark, system (follows OS preference), and cosmos - resolved server-side on first render, no flash.

Architecture

Four services, one event bus

The marketplace publishes domain events to SNS; downstream services consume them via SQS with DLQs. Cross-service discovery happens through SSM Parameter Store - no hardcoded ARNs, services can be torn down and redeployed independently.

Engineering decisions

The choices behind the diagram

Email out of the request path

The Stripe webhook returns 200 as soon as the order is persisted and an SNS event is published. SES failures don't break checkout; SQS retries and a DLQ absorb them. Checkout success and customer notification are decoupled - the way they should be.

Exactly-once email under at-least-once SQS

The notifications Lambda claims each event with a conditional PutItem on NOTIF#{eventId} in DynamoDB, TTL 30 days. SQS may redeliver; the conditional write guarantees one email per event. No second-system bookkeeping, no outbox table on the marketplace side.

Single-table DynamoDB for chat

Conversations, messages, connection records, and reactions all share one table with two GSIs. No cross-table joins. GSI1 answers "list this user's conversations sorted by last message"; GSI2 answers "give me every active WebSocket connection for this user so I can fan-out". A TTL attribute reaps stale connections without a cron.

Cross-service discovery via SSM

No service hardcodes another service's ARN. Each one publishes its SNS topic ARN (and the API keys it accepts) to its own SSM namespace on deploy; consumers resolve them at Lambda cold start and cache for the lifetime of the execution environment. Services can be redeployed independently.

Browser → S3 direct, with safety net

Product and chat-attachment uploads bypass the application server entirely - presigned PUTs from the browser. Every freshly uploaded object carries a lifecycle=pending tag; an S3 lifecycle rule sweeps anything still tagged after 24h, so abandoned form uploads never become long-term orphans.

Cross-language search blob

Product translations live in their own table, but the search index is a denormalized text blob that concatenates the title and description across all locales. Searching for "haljina" surfaces a product whose canonical copy is in English. One blob, one index, no multilingual search engine.

Zero-trust between services

Inter-service calls authenticate with internal API keys stored in SSM SecureString - never in Lambda environment variables, never in source. Each service reads its key at cold start and verifies the x-api-key header on incoming requests.

React 19 + React Compiler

The storefront runs on the experimental React Compiler. No hand-written useMemo or useCallback; memoization is handled at build time. Server components do the heavy lifting; client islands are reserved for cart, filters, and the chat drawer.

Platform-funded coupons, seller made whole

Coupon discounts come out of the platform's commission, not the seller's payout - the seller receives full net on every line, applied as a one-off Stripe discount at checkout (percent / fixed · min-order · usage cap · expiry). Shipping is the seller's: each org sets its own flat rate and free-shipping threshold, totaled per-seller on multi-vendor carts.

Order status on two axes

Order state isn't one enum. PaymentStatus (unpaid → paid → refunded) and FulfillmentStatus (unfulfilled → delivered) move independently and the display status is derived. A COD order can be delivered-but-unpaid; a Stripe order paid-but-unshipped - both representable without contradiction.

Append-only money ledger + audit log

Every money movement - charge, refund, payout, fee - is written to an append-only PaymentTransaction ledger, optionally per seller org, so refunds and Stripe Connect payouts reconcile against a single trail. A separate append-only AuditLog records every admin/seller mutation (actor, action, JSON diff), surfaced read-only in the admin panel.
